# 代码归档规范

## 一、设计目标

代码归档不是简单保存 `.cpp` 文件，而是服务于长期教学分析。

本规范用于统一记录学生在比赛、训练、补题、订正和模板整理过程中的代码，便于后续完成：

```text
1. 追踪学生真实代码能力
2. 保留错误代码与订正过程
3. 分析常见实现问题
4. 关联题目库、训练记录、比赛记录和学生档案
5. 辅助生成阶段学习报告
6. 支持后续针对性训练安排
```

核心原则：

```text
按学生归档
按来源分组
按题目建目录
按版本保存代码
用 review.md 记录复盘
用索引文件做统计
```

```text

```

示例：

```text

```text

## 四、来源编号规范

所有代码必须能追溯来源。

来源分三类：

```text
C：比赛 Contest
T：训练 Training
R：补题 / 订正 Review
```

### 4.1 比赛编号

格式：

```text
C{年份}{序号}-{比赛名称}
```

示例：

```text
C2026-001-春季模拟赛1
C2026-002-前缀和差分测验
C2026-003-暑期综合赛
```

### 4.2 训练编号

格式：

```text
T{日期}-{训练主题}
```

示例：

```text
T2026-05-01-前缀和差分
T2026-05-08-二分与贪心
T2026-05-15-DP入门
```

### 4.3 补题编号

格式：

```text
R{日期}-{来源说明}
```

示例：

```text
R2026-05-03-C2026-001赛后补题
R2026-05-10-前缀和差分二刷
R2026-05-17-课堂订正
```

## 八、代码版本命名规范

代码版本必须同时记录“阶段”和“轮次”。

推荐格式：

```text
{阶段}-v{两位编号}-{结果}.{扩展名}
```

其中 `{阶段}` 推荐使用：

| 阶段     | 含义                                   |
| -------- | -------------------------------------- |
| original | 原始提交 / 赛时提交 / 训练首次提交序列 |
| fixed    | 订正代码序列                           |
| retry    | 补题重做序列                           |
| second   | 二刷独立完成序列                       |
| teacher  | 老师提示后修改序列                     |
| template | 模板整理代码                           |

### 8.1 原始提交版本

```text
original-v01-WA.cpp
original-v02-TLE.cpp
original-v03-AC.cpp
```

说明：

```text
original 表示学生在原始比赛或训练过程中产生的代码。
即使后面已经 AC，也不要覆盖原始错误版本。
```

### 8.2 订正版本

订正代码也可能有多个版本，不能只保存一个 `after-class-fixed.cpp`。

推荐写法：

```text
fixed-v01-WA.cpp
fixed-v02-WA.cpp
fixed-v03-AC.cpp
```

如果要区分订正来源，可以写：

```text
fixed-after-class-v01-WA.cpp
fixed-after-class-v02-AC.cpp
fixed-after-contest-v01-WA.cpp
fixed-after-contest-v02-AC.cpp
```

说明：

```text
fixed 表示学生针对原题进行订正。
订正过程中如果仍然 WA/TLE/RE，也必须保留。
这部分最能反映学生是否真正理解了错因。
```

### 8.3 补题 / 二刷版本

```text
retry-v01-WA.cpp
retry-v02-AC.cpp
second-v01-AC.cpp
```

含义：

| 前缀    | 含义                               |
| ------- | ---------------------------------- |
| retry   | 补题重做，可能参考过题解或老师讲解 |
| second  | 间隔一段时间后的二刷，强调独立完成 |
| teacher | 老师提示后修改，不算完全独立       |

### 8.4 比赛题目前缀

如果比赛中没有按题目库编号归档，而是临时保存，可用：

```text
A-original-v01-WA.cpp
A-original-v02-AC.cpp
B-original-v01-TLE.cpp
B-fixed-v01-WA.cpp
B-fixed-v02-AC.cpp
```

长期归档时建议转为：

```text
A-P0030/original-v01-WA.cpp
A-P0030/original-v02-AC.cpp
B-P0102/original-v01-TLE.cpp
B-P0102/fixed-v01-WA.cpp
B-P0102/fixed-v02-AC.cpp
```

## 九、订正链记录规范

同一道题可能经历多轮订正，因此 `review.md` 中必须单独记录订正链。

### 9.1 订正链字段

建议记录：

```text
原始错误版本
第几次订正
订正依据
订正结果
是否独立完成
是否仍需二刷
```

### 9.2 订正链示例

```markdown
## 订正链

| 阶段 | 文件 | 结果 | 订正依据 | 说明 |
|---|---|---|---|---|
| 原始提交 | original-v01-WA.cpp | WA | 无 | 区间右端点错误 |
| 第一次订正 | fixed-v01-WA.cpp | WA | 看了讲评 | 只修了右端点，漏了 exact 点贡献 |
| 第二次订正 | fixed-v02-AC.cpp | AC | 独立重写 | 完成修正 |
| 二刷 | second-v01-待完成.cpp | 待完成 | 计划安排 | 一周后独立重做 |
```

### 9.3 判断标准

```text
订正 AC 不等于真正掌握。
如果 fixed-v01 仍然 WA，说明学生对错因理解还不完整。
如果 second-v01 能独立 AC，才说明该问题基本消化。
```

## 十一、代码文件头部规范

代码文件头部只记录稳定元信息，不重复维护题目标签和三维难度。

推荐头部：

```cpp
// student: S001
// source: T2026-05-01-前缀和差分
// problem_id: P0102
// platform: CodeForces
// origin_id: 1343D
// version: v01
// submit_type: original
// verdict: WA
// date: 2026-05-13
// note: 区间右端点处理错误
```

字段说明：

| 字段        | 含义                                                   |
| ----------- | ------------------------------------------------------ |
| student     | 学生编号                                               |
| source      | 来源编号                                               |
| problem_id  | 题目库内部编号                                         |
| platform    | 原题平台                                               |
| origin_id   | 原平台题号                                             |
| version     | 当前代码版本，如 original-v01 / fixed-v02              |
| submit_type | original / fixed / retry / second / teacher / template |
| verdict     | 提交结果                                               |
| date        | 日期                                                   |
| note        | 简短备注                                               |

不建议在代码头部写：

```text
完整标签
三维难度
综合等级
详细题解
```

这些信息应以 `02-题目库/题目档案/` 为准，避免重复维护导致不一致。

## 十二、单题 review.md 模板

每个题目代码目录建议配一个 `review.md`。

```markdown
# P0102 代码复盘

## 一、基本信息

| 字段 | 内容 |
|---|---|
| 学生 | S001-张三 |
| 来源 | T2026-05-01-前缀和差分 |
| 题目 | P0102 |
| 原题 | CodeForces 1343D |
| 最终结果 | AC |
| 是否赛时完成 | 否 |
| 是否独立完成 | 部分独立 |
| 是否看题解 | 是 |
| 是否看老师代码 | 否 |

## 二、提交记录

| 版本 | 文件 | 结果 | 问题 |
|---|---|---|---|
| original-v01 | original-v01-WA.cpp | WA | 1 次修改区间右端点错误 |
| fixed-v01 | fixed-v01-WA.cpp | WA | 0 次修改点贡献处理错误 |
| fixed-v02 | fixed-v02-AC.cpp | AC | 修正贡献统计 |

## 三、主要错因

- 差分区间边界不熟
- 没有区分 0 次修改点和 1 次修改区间
- 多组数据清空意识不足

## 四、代码问题

| 类型 | 说明 |
|---|---|
| 边界处理 | r + 1 位置漏处理 |
| 变量命名 | l, r, x 含义不够清晰 |
| 调试习惯 | 只测样例，没有造边界数据 |

## 五、订正情况

- 是否完成订正：是
- 订正版本数：2
- 最终订正结果：AC
- 是否独立重写：否
- 是否二刷：待安排

## 六、后续建议

- 复习差分区间加减
- 重新做 2 道区间贡献题
- 下次写完后必须手推一组边界样例
```

```

单个问题文件模板：

```markdown
# 多组数据未清空问题

## 一、问题描述

学生在多组数据中未清空数组、vector、map 或全局变量，导致后一组数据受到前一组影响。

## 二、典型错误代码

```cpp
while (T--) {
    cin >> n;
    for (int i = 1; i <= n; i++) cin >> a[i];
    // 忘记清空 cnt / vis / vector
}
```

## 三、正确写法

```cpp
while (T--) {
    cin >> n;
    vector<int> a(n + 1);
    map<int, int> cnt;
}
```

## 四、出现记录

| 日期       | 学生 | 来源        | 题目  | 说明            |
| ---------- | ---- | ----------- | ----- | --------------- |
| 2026-05-01 | S001 | T2026-05-01 | P0102 | diff 数组未清空 |
| 2026-05-08 | S003 | C2026-002   | P0096 | map 统计残留    |

## 五、教学处理建议

- 强调局部变量优先
- 多组数据题写完后检查所有状态是否重置
- 训练时专门安排多组数据反例

```

## 十七、索引文件规范

为了后续统计，建议维护索引文件。

### 16.1 学生代码索引.md

```markdown
# 学生代码索引

| 学生 | 来源 | 题目 | 文件 | 结果 | 类型 | 日期 | 问题 |

## 十八、代码归档流程

每次归档代码建议按以下流程：

```text
1. 确定学生编号
2. 确定来源类型：比赛 / 训练 / 补题
3. 确定题目库编号
4. 建立对应题目目录
5. 保存原始代码，不覆盖
6. 按版本命名提交代码
7. 记录 verdict
8. 若有订正，另存 fixed 版本
9. 填写 review.md
10. 更新索引文件
11. 如发现共性问题，补充到共性代码问题目录
```

```text
1. 是否放在正确学生目录？

3. 是否关联题目库 problem_id？
4. 是否保留原始错误代码？
5. 是否没有覆盖旧版本？
6. 文件名是否包含版本和结果？
7. 是否填写 review.md？
8. 是否更新学生代码索引？
9. 是否需要记录共性代码问题？
10. 是否能从代码追溯到比赛/训练/补题来源？
```

## 二十一、最终原则

代码档案最重要的不是保存最终 AC 代码，而是保存学生的真实变化过程。

因此必须做到：

```text
原始代码不能覆盖
错误代码要保留
订正代码要按版本单独存
订正失败版本也要保留
每题要有复盘
每次训练/比赛要能追溯
共性问题要能统计
```

这样代码档案才能真正服务于：

```text
学生画像
错因分析
训练推荐
家长沟通
阶段总结
```
